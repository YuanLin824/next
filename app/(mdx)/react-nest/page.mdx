## OOP

面向对象编程 (Object Oriented Programming)

特点:

1. 抽象事物的特征, 对于理解友好
2. 封装 (高内聚, 低耦合), 继承, 多态
3. 典型的 OOP 语言: Java, C#

## FP

函数式编程 (Function Programming)

1. 对过程进行通用式封装
2. 多处复用封装的函数

特点:

1. 函数是第一等公民
2. 相同的输入必定有相同的输出
3. 纯函数, 相对独立 (无副作用, 即函数内部不会影响到函数外部)
4. 引用透明, 易于理解, 对 IDE 友好
5. vue / react 都偏向 FP

## FRP

函数响应式编程 (Function Response Programming)

特点:

1. 适合需要对事件流进行复杂组合的应用场景
2. 响应式多用在异步场景
3. 典型案例: rxjs, 广告推荐

## AOP

面向切面编程 (Aspect Oriented Programming)

特点:

1. 对 OOP 的补充
2. AOP 在不破坏封装功能的前提下, 额外增加功能
3. 扩展功能方便, 不影响业务之间的逻辑
4. 逻辑集中管理
5. 代码复用

## IOC & DI

IOC: "别叫我，我会叫你" (Don't call us, we'll call you)
DI: "告诉我你要什么, 我给你什么"

控制反转 (Inversion Of Control) 和依赖注入 (Dependency Injection)是现代软件开发的一种设计模式

优点 ✅

1. ✅ 低耦合 - 组件之间松散耦合
2. ✅ 易测试 - 可以注入 mock 对象
3. ✅ 可维护 - 修改实现不影响使用方
4. ✅ 可扩展 - 轻松替换实现
5. ✅ 复用性 - 依赖可在多处复用

缺点 ❌

1. ❌ 学习曲线 - 初学者难理解
2. ❌ 调试困难 - 对象创建过程隐藏
3. ❌ 过度设计 - 小项目可能没必要
4. ❌ 性能开销 - 反射和容器有一定开销

IOC: (设计原则)

1. 控制反转是 OOP 中的一种设计原则, 用于减少代码之间的耦合度
2. 基本思想: 借助于第三方, 实现具有依赖关系对象之间的解耦
3. 把对象的创建和管理控制权交给框架 / 容器

DI: (实现方式)

1. DI 是 IOC 一种的实现方式
2. 允许在类外创建依赖对象, 并通过不同方式将这些对象提供给类
3. DI 的三种注入方式：构造函数注入、属性注入、方法注入
   1. 构造函数注入：通过构造函数传入依赖对象

      ```ts
      // ✅ 依赖明确
      // ✅ 对象创建后立即可用
      // ✅ 支持不可变对象
      class EmailService {
        sendEmail(to: string, content: string) {
          console.log(`发送邮件到 ${to}: ${content}`)
        }
      }
      class UserService {
        // ✅ 通过构造函数注入
        constructor(private emailService: EmailService) {}
        registerUser(email: string) {
          // 业务逻辑
          this.emailService.sendEmail(email, "欢迎注册!")
        }
      }
      // 使用
      const userService = new UserService(new EmailService())
      userService.registerUser("user@example.com")
      ```

   2. 属性注入：通过属性赋值的方式传入依赖对象

      ```ts
      // ⚠️ 依赖不明确
      // ⚠️ 可能忘记注入
      class UserService {
        // ✅ 通过属性注入
        emailService!: EmailService
        registerUser(email: string) {
          this.emailService.sendEmail(email, "欢迎注册!")
        }
      }
      // 使用
      const userService = new UserService()
      userService.emailService = new EmailService() // 注入依赖
      userService.registerUser("user@example.com")
      ```

   3. 方法注入：通过方法参数的方式传入依赖对象

      ```ts
      class UserService {
        // ✅ 通过方法参数注入
        registerUser(email: string, emailService: EmailService) {
          emailService.sendEmail(email, "欢迎注册!")
        }
      }
      // 使用
      const userService = new UserService()
      userService.registerUser("user@example.com", new EmailService())
      ```

```ts
class Phone {
  playGame(name: string) {
    console.log(name)
  }
}
// Person  => play => Phone 的强依赖关系
class Person {
  constructor(private name: string) {
    this.name = name
  }

  play() {
    const phone = new Phone()
    phone.playGame(this.name)
  }
}

// 实现依赖注入的思想就是控制反转
// 通过依赖注入的方式解耦 Person 和 Phone 的强依赖关系
// 类似函数传参
interface InterfacePhone {
  playGame(name: string): void
}
class IPhone implements InterfacePhone {
  playGame(name: string): void {
    console.log(`${name} is playing game on IPhone`)
  }
}
class Android implements InterfacePhone {
  playGame(name: string): void {
    console.log(`${name} is playing game on Android`)
  }
}
class DIPerson {
  constructor(
    private name: string,
    private phone: InterfacePhone
  ) {
    this.name = name
    this.phone = phone
  }

  play() {
    this.phone.playGame(this.name)
  }
}

// new IPhone() 和 new Android() 就是依赖注入
const p1 = new DIPerson("Tom", new IPhone())
p1.play() // Tom is playing game on IPhone
const p2 = new DIPerson("Jerry", new Android())
p2.play() // Jerry is playing game on Android
```

## MVC

MVC (Model-View-Controller) 是一种软件架构模式, 广泛应用于开发 Web 应用程序

1. 它将应用程序分为三个主要组件:
   1. 模型
   2. 视图
   3. 控制器
2. NestJS 实现 View 层:
   1. pug
   2. hus
   3. ejs
   4. ......
3. NestJS 默认继承 Express 作为 Web 服务器, 可以换成 Koa, Fastify...
4. Controller 响应前端的请求, Model 是对应的具体数据库逻辑

## DTO & DAO

请求 => DTO \<=\> 逻辑 \<=\> DAO => 数据库

数据传输对象 (Date Transfer Object):

1. 接受部分数据
2. 对数据进行筛选
3. 不对应实体
4. 属性小于等于实体

数据访问对象 (Date Acces Object):

1. 对受数据库接口
2. 不暴露数据库的内部信息
3. 对应实体

NestJS 中的 DTO:

1. 约定了数据字段, 属性
2. 方便数据校验

NestJS 中的实体类:

1. DAO 是一层逻辑, 包含:
   1. 实体类
   2. 数据库操作 (CURD)
   3. 数据库校验
   4. 错误处理等
2. NestJS 做了一层封装:
   1. 通过 ORM 库与数据库对接 (通过 ORM 可以对接多种数据库)
   2. 这些 ORM 库就是 DAO 层

中间件面向切面: 不改变原逻辑, 为系统提供个性化处理

- 系统日志
- 操作日志
- 鉴权 (守卫)

## Class Middleware

### src/common/middlewares/log.middleware.ts

```ts
import { Inject, Injectable, Logger, NestMiddleware } from "@nestjs/common"
import { HttpAdapterHost } from "@nestjs/core"
import { NextFunction, Request, Response } from "express"
import requestIp from "request-ip"

@Injectable()
export class LogMiddleware implements NestMiddleware {
  private readonly logger = new Logger(LogMiddleware.name)

  constructor(private readonly httpAdapterHost: HttpAdapterHost) {}

  use(req: Request, _res: Response, next: NextFunction) {
    const { httpAdapter } = this.httpAdapterHost

    // 异步写入日志，不 await —— 即使日志写入失败也不影响正常请求
    logger({
      clientIp: requestIp.getClientIp(req) ?? "unknown",
      method: httpAdapter.getRequestMethod(req).toLocaleUpperCase(),
      path: httpAdapter.getRequestUrl(req),
      body: req.body ?? {},
      params: req.params ?? {},
      query: req.query ?? {},
    })
    next()
  }
}
```

## src/common/exceptions/all-exception.filter.ts

```ts
import {
  ArgumentsHost,
  Catch,
  ExceptionFilter,
  HttpException,
  HttpStatus,
  Logger,
} from "@nestjs/common"
import { HttpAdapterHost } from "@nestjs/core"
import requestIp from "request-ip"
import { QueryFailedError } from "typeorm"

const FALLBACK_DB_CODE = "__fallback__"

/**
 * PostgreSQL 错误码 → HTTP 状态码 & 中文消息
 *
 * Postgres 错误码分类：
 *   Class 23 — 完整性约束（唯一、外键、非空、检查）
 *   Class 22 — 数据异常（值超长、类型不匹配）
 *   Class 40 — 事务/死锁
 *
 * 将数据库底层错误翻译为前端可读的中文消息，避免暴露数据库细节
 */
const DB_ERROR_MAP: Record<string, { status: number; message: string }> = {
  // 完整性约束 (Class 23)
  "23505": { status: HttpStatus.CONFLICT, message: "数据已存在, 违反唯一约束" },
  "23503": { status: HttpStatus.BAD_REQUEST, message: "数据关联异常, 违反外键约束" },
  "23502": { status: HttpStatus.BAD_REQUEST, message: "必填字段缺失, 违反非空约束" },
  "23514": { status: HttpStatus.BAD_REQUEST, message: "数据校验失败, 违反检查约束" },
  "23506": { status: HttpStatus.CONFLICT, message: "违反排他约束" },
  // 数据异常 (Class 22)
  "22001": { status: HttpStatus.BAD_REQUEST, message: "字段值超出最大长度限制" },
  "22P02": { status: HttpStatus.BAD_REQUEST, message: "字段类型不匹配, 数据格式无效" },
  // 死锁 (Class 40)
  "40P01": { status: HttpStatus.CONFLICT, message: "检测到死锁, 请重试" },
  // 兜底
  [FALLBACK_DB_CODE]: { status: HttpStatus.INTERNAL_SERVER_ERROR, message: "数据库操作异常" },
}

/**
 * 全局异常过滤器
 *
 * 统一处理三类异常：
 * 1. HttpException — NestJS 内置及自定义业务异常，直接提取状态码与消息
 * 2. QueryFailedError — TypeORM 数据库异常，通过 Postgres 错误码映射为友好中文提示
 * 3. 其他未知异常 — 兜底返回 500
 *
 * 所有异常均记录结构化日志（含客户端 IP、路径、方法、堆栈），
 * 响应格式统一为 { code, data, message }
 */
@Catch()
export class AllExceptionFilter<T> implements ExceptionFilter {
  private readonly logger = new Logger(AllExceptionFilter.name)

  constructor(private readonly httpAdapterHost: HttpAdapterHost) {}

  catch(exception: T, host: ArgumentsHost) {
    const { httpAdapter } = this.httpAdapterHost
    const ctx = host.switchToHttp()
    const res = ctx.getResponse()
    const req = ctx.getRequest()
    const clientIp = requestIp.getClientIp(req)

    let status = HttpStatus.INTERNAL_SERVER_ERROR
    let message = "Internal Server Error"

    if (exception instanceof HttpException) {
      status = exception.getStatus()
      message = exception.message
    }

    // 将 Postgres 错误码翻译为可读的中文消息
    if (exception instanceof QueryFailedError) {
      const driverError = exception.driverError as {
        code?: string
        detail?: string
        table?: string
      }
      const pgCode = driverError?.code ?? FALLBACK_DB_CODE
      const mapping = DB_ERROR_MAP[pgCode] ?? DB_ERROR_MAP[FALLBACK_DB_CODE]
      status = mapping?.status
      message = mapping?.message
    }

    this.logger.error({
      clientIp,
      method: httpAdapter.getRequestMethod(req),
      path: httpAdapter.getRequestUrl(req),
      code: status,
      message: message,
      stack: exception,
      timestamp: new Date().toISOString(),
    })

    let responseData: Record<string, unknown> = {}
    if (exception instanceof HttpException) {
      const exceptionResponse = exception.getResponse()
      if (typeof exceptionResponse === "object" && exceptionResponse !== null) {
        responseData = exceptionResponse as Record<string, unknown>
      }
    }
    httpAdapter.reply(res, { code: status, data: responseData, message: message }, status)
  }
}
```

# Log

1. 日志
   1. 什么时间发生的
   2. 发生了什么
   3. 错误信息

   1. 控制台日志: 方便监看
   2. 文件日志: 方便回溯与追踪
   3. 数据库日志: 敏感操作, 敏感数据记录

### Winston Logger 配置

```ts
// ./src/config/winston.ts
import { utilities, WinstonModule } from "nest-winston"
import winston from "winston"
import DailyRotateFile from "winston-daily-rotate-file"
import { IS_PROD } from "../common/utils/global"

/** 生产环境: 按小时轮转的日志文件，压缩归档，自动清理 14 天前的日志 */
const prod_opts = {
  dirname: "logger_prod",
  datePattern: "YYYY-MM-DD-HH",
  zippedArchive: true,
  maxSize: "20m",
  maxFiles: "14d",
  format: winston.format.combine(
    winston.format.timestamp(),
    utilities.format.nestLike("NestApp", { prettyPrint: true, colors: false })
  ),
}

/** 开发环境: 彩色控制台输出 */
const dev_opts = {
  format: winston.format.combine(winston.format.timestamp(), utilities.format.nestLike(undefined)),
}

/**
 * 创建 NestJS 兼容的 Winston Logger 实例
 *
 * 供 NestFactory.create 使用，替换 NestJS 默认的控制台 Logger。
 * 生产环境输出到 info/error 两个文件，开发环境输出到控制台。
 */
export const WINSTON_LOGGER = WinstonModule.createLogger({
  transports: IS_PROD
    ? [
        new DailyRotateFile({ level: "info", filename: "%DATE%.info.log", ...prod_opts }),
        new DailyRotateFile({ level: "error", filename: "%DATE%.error.log", ...prod_opts }),
      ]
    : [new winston.transports.Console({ level: "info", ...dev_opts })],
})
```

### 使用 Winston Logger 替换内置 Logger

```ts
import { NestFactory } from "@nestjs/core"
import { NestExpressApplication } from "@nestjs/platform-express"
import { AppModule } from "./app.module"
import { WINSTON_LOGGER } from "./config/winston"

async function bootstrap() {
  // 使用 Winston 日志替代默认 Logger，确保启动阶段的日志也能输出到文件
  const app = await NestFactory.create<NestExpressApplication>(AppModule, {
    logger: WINSTON_LOGGER,
  })

  await app.listen(3000)
}

bootstrap()
  .then(() => {
    console.log(`应用启动成功! 接口 URL:`, `http://localhost:3000`)
  })
  .catch((err) => {
    console.error("应用启动失败:", err)
    process.exit(1)
  })
```

SWC 编译 src 外的文件:

1. NestJS 中的 SWC 默认配置只会编译 src 目录下的文件
1. 配置 .swcrc 文件的 include
1. nest-cli.json 的 compilerOptions.builder.options.swcrcPath 项正确指向 .swcrc

npm install --save-dev @swc/cli @swc/core chokidar

# ORM

1. 是什么:
   1. ORM (Object Relational Mappine) 对象映射关系
   2. 作用: 把面向对象的概念跟数据库中的概念对应起来
   3. 示例: 定义一个对象, 对应着一张表; 对象实例, 对应着表中的一条记录
2. 优点:
   1. 方便维护: 数据模型定义在同一个地方, 利于重构
   2. 代码量少, 对接多种数据库: 代码逻辑通俗易懂
   3. 工具多, 自动化能力强: 数据库删除, 关联数据, 事务操作等
3. 缺点:
   1. 不能优化 SQL 语句
4. 推荐库:
   1. knex
   2. prisma
   3. sequelize
   4. typeorm

## 分布式锁

1. 设计与实现
   1. 分布式系统中, 为了防止多个服务实例同时访问共享资源, 导致数据不一致的问题, 分布式锁是必不可少的
2. 常见场景
   1. 库存扣减: 高并发秒杀场景, 确保商品不超卖
   2. 定时任务: 多个实例运行时, 确保只有一个实例执行任务
   3. 资源更新: 多个服务实例对同一资源进行更新时, 避免冲突
3. 技术实现
   1. 使用 Redis 作为分布式工具
   2. 基于 Redis 的 SET 命令, 结合 NX 和 EX 参数实现加锁
4. Redis 分布式锁核心命令
   1. SET lock_key unique_value NX EX 10 # 加锁
   2. DEL lock_key # 释放锁
   3. NX: 表示只有当 locak_key 不存在时才设置成功
   4. EX: 设置过期时间, 防止锁死
   5. NestJS 实现分布式锁 (使用 Redis 实现)
      1. Redis 模块: 创建全局 Redis 模块, 用于连接 Redis
      2. 分布式锁服务: 封装分布式锁的核心逻辑
      3. 使用分布式锁
5. 注意事项:
   1. 锁的唯一性: 锁的只 (unique_value) 需要全局唯一, 防止其他服务误删
   2. 锁的过期时间: 设置合理的过期时间, 防止锁死
   3. 重试机制: 在获取锁失败时, 需要有重试机制
   4. 高性能优化: Redis 是单线程模型, 过多的锁竞争会影响性能, 可使用分布式库 (如 Redlock) 来优化锁

## 架构设计一个复杂业务的后端服务 (数据持久化最佳实践)

### 复杂项目服务端架构设计重点

1. 复杂业务场景下, 服务端的架构设计核心需求:
   1. 高可用: 系统能够持续稳定地运行, 即使在部分组件出现故障时也能保证服务的正常运行
   2. 高性能: 快速响应用户请求, 支持大量用户同时访问
   3. 可扩展: 系统能够轻松扩展以支持更大的负载
   4. 可维护: 代码结构清晰, 逻辑分离, 方便开发和调试
   5. 数据一致: 分布式环境中确保事务一致性
   6. 数据容灾 (数据安全性): 支持数据备份, 容灾和加密, 快速恢复服务, 减少故障对用户的影响
2. 关键设计问题 (并发):
   1. 缓存设计: 快速访问频繁使用的数据, 减轻数据库压力
   2. 数据库设计: 选择合适的数据库, 根据场景调整表结构和查询优化
   3. 日志数据库设计: 集中式日志存储与分析, 用于调试和性能监控
   4. 海量数据设计: 优化数据存储和查询, 降低延迟和存储成本
   5. 实时数据流设计: 处理实时事件流和数据流计算

## 服务端高可用, 高并发实战经验 (以 Nodejs 服务端架构为例)

### 最佳实践总结

1.  分布式锁: 使用 Redis 分布式锁解决高并发下的资源竞争问题
2.  读写分离: 使用 PostgreSQL 主从复制实现读写分离, 提高性能
3.  服务解耦: 通过 Kafka 实现异步通信, 降低服务之间的耦合度
4.  缓存层优化: 优先缓存热点数据, 避免频繁访问数据库
5.  容器化部署: 使用 Docker 和 K8s 进行服务管理, 实现自动扩容和故障恢复

### 高可用, 高并发实战经验

1. 微服务架构设计:
   1. 核心思想:
      1. 将系统拆分为多个独立的服务, 每个服务负责特定的业务功能
      2. 服务之间通过异步消息或 HTTP 通信 (松耦合, 易扩展)
   2. 优势:
      1. 独立部署, 增强可靠性: 每个服务可以独立部署, 快速迭代
      2. 横向扩展灵活, 资源利用率高: 根据需求增加或减少服务实例
2. 分布式系统设计:
   1. 分布式消息队列: 使用 Kafka 处理异步任务. 解耦服务之间的依赖
   2. 分布式数据库: 实现分片存储与主从复制
3. 技术选型:
   1. NestJS: 提供模块化架构, 支持微服务开发
   2. Kafka: 分布式消息队列, 支持高吞吐和低延迟
   3. MySQL: 关系型数据库, 支持事务和外键约束
   4. Redis: 提供分布式缓存与分布式锁
   5. Docker + K8s: 服务容器化和编排管理

### 高可用, 高并发设计思路

1. 网关层:
   1. 使用 API Gateway (如: Nginx, NestJS Gateway) 处理所有外部请求
   2. 实现功能:
      1. 均衡负载: 将请求分发到多个服务实例
